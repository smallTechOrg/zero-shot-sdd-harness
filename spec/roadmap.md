# Roadmap

## What This Agent Does

A conversational data analysis assistant that lets users upload CSV files and ask natural-language questions in a multi-turn session. The agent profiles uploaded data automatically, generates Python/pandas code server-side, executes it in a sandboxed environment, and returns answers as plain prose, summary tables, and interactive Plotly charts embedded in the chat. Raw row values never leave the server — the LLM receives only schema metadata and statistical summaries.

## Who Uses It

Personal daily analysts and non-technical stakeholders who need to explore tabular data without writing code. They upload a file, read the auto-generated profile card to understand the data, then ask questions in plain English across multiple turns in the same session.

## Core Problem Being Solved

Non-technical users cannot query CSV data without writing SQL or Python. Existing tools either require coding skill or send raw data to external services. This agent answers data questions conversationally, executes code locally for privacy, and returns visual answers (charts, tables, prose) in one unified interface.

## Success Criteria

- [ ] A user can upload a CSV and receive a profile card (row count, column types, null counts, sample values, quality flags) within 5 seconds.
- [ ] A user can ask a natural-language question and receive a text answer plus an interactive Plotly chart within 15 seconds.
- [ ] Conversation history is preserved within a session: a follow-up question referencing "that column" resolves correctly using prior context.
- [ ] Raw data row values are provably absent from every LLM prompt (verified by test that inspects prompt contents).
- [ ] The E2E Playwright test completes the full upload → profile → Q&A path without manual intervention.

## What This Agent Does NOT Do (Out of Scope)

- Persist user data or conversation history across server restarts or sessions — session-only.
- Accept Excel (.xlsx) files in Phase 1 — CSV only until Phase 2.
- Join or compare multiple files in Phase 1 — single-file only until Phase 2.
- Export results to PDF, Excel, or CSV in Phase 1.
- Send raw row values to the LLM at any phase — architectural constraint, not a deferral.
- Authenticate users or enforce access control — single-user personal tool.
- Write data back to the uploaded file.
- Train or fine-tune any model.

## Key Constraints

- **Privacy**: The LLM receives only column names, dtypes, statistical summaries (min/max/mean/std/quantiles for numerics, value_counts top-10 for categoricals), and null counts. Zero raw row values.
- **Session-only**: Uploaded files are stored in a per-session temp directory deleted on session end or server restart. SQLite holds only session metadata and message history — no file content.
- **LLM cost**: One LLM call per user question (plan_and_code) + one LLM call per answer (format_response). Profile is deterministic, no LLM. Conversation history trimmed to last 20 turns before sending.
- **Code execution**: pandas code generated by the LLM is executed in a sandboxed `exec()` call with a restricted globals dict. Execution timeout: 30 seconds.
- **Gemini**: `google-genai` SDK, `AGENT_GEMINI_API_KEY` from `.env`. Default model: `gemini-3.1-pro` (env-configurable via `AGENT_LLM_MODEL`).
- **Charts**: Plotly JSON spec returned from server, rendered client-side with plotly.js — never PNG.

## Phases of Development

### Phase 1 — End-to-End Core Path

- **Goal:** A user uploads a single CSV, sees a profile summary card (row count, column types, nulls, sample values, quality flags), asks one natural-language question, and receives a plain-text answer with an embedded interactive Plotly chart. The full pipeline works end-to-end, first-time-right.

- **Independent slices (parallel build units):**
  - `slice-a` (backend) — Session creation endpoint, CSV upload endpoint, data profiling node (pandas, no LLM), LangGraph graph wiring (all four nodes: `profile_data`, `plan_and_code`, `execute_code`, `format_response`; all complete, not stubs), Q&A endpoint, sandboxed code execution engine, response formatting, SQLite schema + Alembic migrations, structured logging + LangSmith tracing. No dependency on slice-b.
  - `slice-b` (frontend) — Upload dropzone UI, profile summary card component, chat interface with message history, Plotly chart rendering (react-plotly.js), spinner/loading states, clearly-labelled non-functional stubs for multi-file and export (Phase 2). No dependency on slice-a at build time; calls the API at runtime.

- **Key surfaces / files:**
  - slice-a: `src/api/routes/sessions.py`, `src/api/routes/files.py`, `src/api/routes/messages.py`, `src/graph/state.py`, `src/graph/nodes.py`, `src/graph/edges.py`, `src/graph/agent.py`, `src/graph/runner.py`, `src/db/models.py`, `alembic/versions/`, `src/prompts/code_gen.md`, `src/prompts/format_response.md`, `tests/integration/test_pipeline.py`
  - slice-b: `frontend/src/app/page.tsx`, `frontend/src/components/FileUpload.tsx`, `frontend/src/components/ProfileCard.tsx`, `frontend/src/components/ChatMessage.tsx`, `frontend/src/components/ChatInput.tsx`, `frontend/src/components/PlotlyChart.tsx`, `frontend/src/lib/api.ts`, `tests/e2e/phase1.spec.ts`

- **Gate command:**
  ```
  uv run alembic upgrade head && uv run pytest tests/integration/test_pipeline.py -v && cd frontend && pnpm build && cd .. && uv run python -m src & sleep 5 && npx playwright test tests/e2e/phase1.spec.ts --reporter=line
  ```

- **How the user tests it:**
  1. Run `cd frontend && pnpm build && cd ..` to build the frontend static export.
  2. Run `uv run python -m src` to start the server (port 8001).
  3. Open `http://localhost:8001/app/` in the browser.
  4. Drag-and-drop any CSV file (e.g., a 500-row sales CSV) onto the upload area.
  5. Expect: within 5 seconds, a profile card appears showing row count, column names with dtypes, null counts per column, top sample values, and quality flags (e.g., "3 duplicate rows detected").
  6. Type a question in the chat input, e.g., "What is the average revenue by region?"
  7. Expect: within 15 seconds, a text answer appears followed by an interactive Plotly bar chart embedded in the chat.
  8. Type a follow-up: "Which region had the highest value?" — expect the agent to resolve context from prior turn.
  - **Real on tested path:** session creation, upload, profile (deterministic), LangGraph Q&A pipeline, code execution, chart rendering.
  - **Labelled stubs (visible but non-functional):** "Upload another file" button (disabled, labelled "Coming in Phase 2"), "Export results" button (disabled, labelled "Coming in Phase 2"), Excel upload note in dropzone ("CSV only — Excel support coming in Phase 2").

### Phase 2 — Multi-File Analysis, Excel Support, and Export

- **Goal:** Users can upload multiple CSV or Excel files in a session, ask questions that join or compare data across files, and export answers as CSV.

- **Independent slices (parallel build units):**
  - `slice-a` (backend — multi-file + Excel) — Multi-file session logic (all uploaded DataFrames available in exec context by filename), Excel reader (openpyxl), updated profiler for multiple dataframes, cross-file query support in code execution context. No dependency on slice-b/c/d.
  - `slice-b` (backend — export) — Export endpoint: generate CSV download from last result table. No dependency on slice-a/c/d.
  - `slice-c` (frontend — multi-file UI) — "Add another file" button wired (enables multi-upload), file list with per-file profile cards and remove buttons, cross-file query hints in chat placeholder. No dependency on slice-a/b/d.
  - `slice-d` (frontend — export UI) — "Export CSV" button wired to export endpoint, browser download flow. No dependency on slice-a/b/c.

- **Key surfaces / files:**
  - slice-a: `src/profiling/profiler.py` (multi-df support), `src/graph/nodes.py` (multi-df exec context), `src/api/routes/files.py` (multi-upload), `tests/integration/test_multifile.py`
  - slice-b: `src/api/routes/export.py`, `tests/integration/test_export.py`
  - slice-c: `frontend/src/components/FileList.tsx`, `frontend/src/app/page.tsx` (stubs wired), `tests/e2e/phase2_multifile.spec.ts`
  - slice-d: `frontend/src/components/ExportMenu.tsx`, `tests/e2e/phase2_export.spec.ts`

- **Gate command:**
  ```
  uv run pytest tests/integration/test_multifile.py tests/integration/test_export.py -v && cd frontend && pnpm build && cd .. && uv run python -m src & sleep 5 && npx playwright test tests/e2e/phase2_multifile.spec.ts tests/e2e/phase2_export.spec.ts --reporter=line
  ```

- **How the user tests it:**
  1. Start server: `uv run python -m src`, open `http://localhost:8001/app/`.
  2. Upload two CSV files (e.g., orders.csv and products.csv). Expect two profile cards.
  3. Ask: "Join orders and products on product_id and show me total revenue by category." Expect text answer and Plotly chart.
  4. Upload an Excel (.xlsx) file. Expect a profile card identical to CSV flow.
  5. Click "Export CSV" after a Q&A result — browser downloads a file with the result table.

### Phase 3 — Agentic Clarification, Reflection, and Error Recovery

- **Goal:** The agent proactively asks clarifying questions when a query is ambiguous, reflects on its own generated code if execution fails (up to 2 retries with corrected code), and surfaces structured error explanations to the user instead of raw tracebacks.

- **Independent slices (parallel build units):**
  - `slice-a` (backend — clarification flow) — Clarification detection node added to graph, new `NEEDS_CLARIFICATION` graph branch, clarification stored as assistant turn, follow-up merges with original question. No dependency on slice-b/c.
  - `slice-b` (backend — reflection + retry) — Reflection node that inspects execution errors and re-generates code (max 2 retries), structured error formatter. No dependency on slice-a/c.
  - `slice-c` (frontend — clarification UI) — Renders clarification questions as assistant bubbles with a highlighted border; user reply treated as follow-up turn. No dependency on slice-a/b.

- **Key surfaces / files:**
  - slice-a: `src/graph/nodes.py` (clarify node), `src/graph/agent.py` (new branch + edge), `tests/integration/test_clarification.py`
  - slice-b: `src/graph/nodes.py` (reflect node), `src/graph/runner.py` (retry logic), `tests/integration/test_reflection.py`
  - slice-c: `frontend/src/components/ClarificationBubble.tsx`, `tests/e2e/phase3_clarification.spec.ts`

- **Gate command:**
  ```
  uv run pytest tests/integration/test_clarification.py tests/integration/test_reflection.py -v && cd frontend && pnpm build && cd .. && uv run python -m src & sleep 5 && npx playwright test tests/e2e/phase3_clarification.spec.ts --reporter=line
  ```

- **How the user tests it:**
  1. Start server: `uv run python -m src`, open `http://localhost:8001/app/`.
  2. Upload a CSV with ambiguous column names (e.g., "val1", "val2").
  3. Ask: "Show me the trend." — expect agent responds: "I see columns val1 and val2 — which one should I plot as the trend?"
  4. Reply with "val1" — expect a Plotly line chart.
  5. Ask a malformed analytical question — expect the agent to retry and return a corrected answer, or surface a clear human-readable error (not a raw traceback).
