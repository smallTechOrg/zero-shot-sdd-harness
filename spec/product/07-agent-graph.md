# Agent Graph

> **Mandatory pre-coding checklist** — answer all questions before writing any node code.
> If any are missing, raise a blocker. (See `spec/engineering/ai-agents.md` Section 10.)
>
> 1. **Action type:** What does the LLM generate? → An MCP tool call: `{"tool": "<table>__run_query", "arguments": {"query": "<SQL>"}}`
> 2. **Termination signal:** How does the LLM say it's done? → Response starts with `FINAL ANSWER:`
> 3. **Recoverable vs fatal errors:** DuckDB SQL errors and non-SELECT SQL → recoverable (the MCP tool returns `isError=True`, fed back to the LLM). Unknown tool name → recoverable. LLM-call failure, missing Parquet, MCP server/session failure → fatal.
> 4. **Max iterations default:** `10` (configurable via `DATAANALYSIS_MAX_AGENT_ITERATIONS`)
> 5. **In-session resource:** A per-`run_id` pool (`graph/mcp_pool.py`) holds one built MCP server (+ its DuckDB connection) per source **across nodes**; `ClientSession`s are **transient** — opened and closed within a single node. The pool's DuckDB connections are closed in `finalize`/`handle_error` (+ `try/finally` backstop in `run_pipeline`).
> 6. **State fields for history:** `action_history: list[dict]`, `iteration_count: int`, `llm_response: str`, `tools: list[dict]` (from `list_tools()` at run start).
> 7. **Tools the agent has:** Per source, one MCP tool, key `<table_name>__run_query`, `inputSchema` auto-generated by FastMCP, description = the source's `capability_description`.
> 8. **Exact tool invocation format:** the JSON above (`tool` + `arguments`).
> 9. **What triggers tool registration:** file upload → a `DataSource` row; the source's MCP server (and its tool) is built at run start from `parquet_path`.

## State

```python
class AgentState(TypedDict, total=False):
    # Identity
    run_id: str
    query_record_id: str
    session_id: str

    # Input
    question: str

    # Tools (aggregated from list_tools() across all source sessions, by load_data)
    tools: list[dict]  # [{"name": "<table>__run_query", "table_name": str,
                       #   "description": str, "parameter_schema": dict}]  (FLAT — no nested capabilities)

    # Schema (derived from DataSource rows by load_data — no DuckDB read needed)
    column_names: list[str]   # "table.column" across all attached sources
    row_count: int

    # ReAct loop state
    action_history: list[dict]   # [{"tool": str, "arguments": dict, "result": str, "is_error": bool}]
    iteration_count: int
    llm_response: str            # raw LLM output from plan_action

    # Final output
    answer: str

    # Usage tracking
    input_tokens: int
    output_tokens: int
    total_tokens: int
    estimated_cost_usd: float | None
    api_request_count: int

    # Control
    error: str | None
```

> The MCP servers and their DuckDB connections are **not** serialisable and do **not** live in state. They live in a module-level pool keyed by `run_id` (mirroring the previous `data_cache` pattern), accessed via `open_pool` / `get_pool` / `close_pool` in `graph/mcp_pool.py`. `ClientSession`s are transient — opened and closed within a single node, never held across nodes.

## Async lifecycle (read before touching nodes)

LangGraph runs **each node in its own asyncio task** (verified empirically). AnyIO ties an async context manager's cancel scope to the task that entered it, so a `ClientSession` (an async CM) **cannot be opened in one node and closed in another** — that raises `RuntimeError: Attempted to exit cancel scope in a different task`. The design therefore:

- All 5 nodes are `async def`. `run_pipeline()` (sync) drives them with `asyncio.run(agent_graph.ainvoke(initial))` on the existing daemon thread.
- The per-run **pool** (`graph/mcp_pool.py`, keyed by `run_id`) holds only **plain objects** that are safe to share across tasks: one built `FastMCP` server (and its DuckDB connection) per attached source. It does **not** hold open `ClientSession`s.
- Each MCP operation opens a **transient** in-memory `ClientSession` and closes it **within a single node** (one task): `load_data` opens a session per server to `list_tools()`; `execute_action` opens a session to `call_tool()`. The same server object backs many sequential sessions.
- `finalize`/`handle_error` call `await close_pool(run_id)` to close the DuckDB connections (a plain, task-safe `conn.close()`); `run_pipeline` wraps `ainvoke` in `try/finally: await close_pool(run_id)` as a backstop.
- **Do not** add parallel fan-out nodes or otherwise span an MCP `ClientSession` across nodes. The sync LLM `complete()` and SQLAlchemy calls are made **directly** inside the async nodes (the dedicated thread makes blocking harmless).

## Nodes

### `load_data`

**Reads from state:** `session_id`, `run_id`

**Writes to state:** `tools`, `column_names`, `row_count`, `action_history=[]`, `iteration_count=0`

**External calls:**

| System | Operation | On Failure |
|--------|-----------|------------|
| SQLite (DB) | Load the session's `DataSource` rows (incl. descriptions) | Fatal |
| MCP (in-process) | Build a `FastMCP` per source + open a `ClientSession`; `list_tools()` | Fatal |

**Behaviour:** Loads the session's attached sources from SQLite. For each, computes `table_name = sql_table_name(name)` and stashes it on the source dict. Calls `await open_pool(run_id, sources)`, which builds one DuckDB-backed `FastMCP` per source over its `parquet_path` (a missing Parquet raises → fatal) and discovers each server's tools via a transient session. Sets `state["tools"] = pool.list_tools()` (the descriptors captured at open). Derives `column_names` (`table.column`) and `row_count` from the `DataSource` rows (no DuckDB read). Sets `error` if the session has no sources.

---

### `plan_action`

**Reads from state:** `question`, `column_names`, `tools`, `action_history`, `iteration_count`, token fields

**Writes to state:** `llm_response`, `answer` (if FINAL ANSWER detected), token fields

**External calls:**

| System | Operation | On Failure |
|--------|-----------|------------|
| OpenRouter (or stub) | Chat completion (sync `complete()`, called directly) | Fatal (close pool) |

**Behaviour:** Builds a prompt containing the available tools (namespaced names + descriptions + parameter schema), the dataset schema, the user's question, and the full `action_history`. Sends to the LLM; accumulates token counts. If the response starts with `FINAL ANSWER:`, strips the prefix, sets `answer`, routes to `finalize`. Otherwise stores `llm_response` and routes to `execute_action`. Stub tag: `<node:plan_action>`.

---

### `execute_action`

**Reads from state:** `run_id`, `llm_response`, `action_history`, `iteration_count`, `tools`

**Writes to state:** `action_history` (appended), `iteration_count` (incremented)

**External calls:**

| System | Operation | On Failure |
|--------|-----------|------------|
| MCP (in-process) | `await get_pool(run_id).call_tool(tool_key, arguments)` → DuckDB SELECT | SQL/unknown-tool = recoverable; missing pool = fatal |

**Behaviour:**
1. Parses `llm_response` as JSON `{"tool": str, "arguments": dict}` (markdown fences stripped). Bad JSON → recoverable observation, loop back.
2. `text, is_error = await get_pool(run_id).call_tool(tool, arguments)`. The pool routes the namespaced key to the owning server, opens a transient session, calls `call_tool`, closes it, and reads `CallToolResult.content[0].text` and `.isError`. Unknown tool key → recoverable message from the pool.
3. Appends `{"tool", "arguments", "result": text, "is_error"}` to `action_history`, increments `iteration_count`.
4. If `iteration_count >= max_iterations` → set `error` (the router sends it to `handle_error`).

---

### `finalize`

**Reads from state:** `run_id`, `query_record_id`, `answer`, `iteration_count`, `action_history`, token fields

**External calls:** SQLite — update QueryRecord (answer, iteration_count, query_history_json, tokens, status=completed) and AgentRun (status=completed). Then `await close_pool(run_id)`.

---

### `handle_error`

**Reads from state:** `error`, `run_id`, `query_record_id`

**External calls:** SQLite — update QueryRecord/AgentRun to `failed` (best-effort). Always `await close_pool(run_id)` in a `finally`. Logs with `run_id`. Terminates the graph.

## Edge Topology

```
START
  │
  ▼
load_data ──(error)──────────────────────────► handle_error ──► END
  │
  ▼
plan_action ◄─────────────────────────────────┐
  │                                           │
  ├──(error)──► handle_error                  │
  │                                           │
  ├──(FINAL ANSWER)──► finalize ──► END       │
  │                                           │
  └──(tool call)──► execute_action ───────────┘
                        │
                        ├──(fatal error)──► handle_error
                        │
                        └──(iteration_count >= max)──► handle_error
```

## Graph Assembly

```python
graph = StateGraph(AgentState)

graph.add_node("load_data", load_data)            # all async def
graph.add_node("plan_action", plan_action)
graph.add_node("execute_action", execute_action)
graph.add_node("finalize", finalize)
graph.add_node("handle_error", handle_error)

graph.set_entry_point("load_data")

graph.add_conditional_edges("load_data", route_after_load,
    {"plan_action": "plan_action", "handle_error": "handle_error"})
graph.add_conditional_edges("plan_action", route_after_plan,
    {"execute_action": "execute_action", "finalize": "finalize", "handle_error": "handle_error"})
graph.add_conditional_edges("execute_action", route_after_execute,
    {"plan_action": "plan_action", "handle_error": "handle_error"})
graph.add_conditional_edges("finalize", route_after_finalize,
    {"end": END, "handle_error": "handle_error"})
graph.add_edge("handle_error", END)

compiled_graph = graph.compile()   # driven via `await compiled_graph.ainvoke(...)`
```

Routing functions stay **sync** (LangGraph conditional-edge routers inspect state only).

## Tool Prompt Format

The `plan_action` prompt describes the MCP tools to the LLM:

```
Available tools (call a tool by its exact name):

Tool: ds_2024_sales__run_query  (queries table: ds_2024_sales)
  Description: Run a read-only SQL SELECT against the 2024 sales dataset.
  Parameters: {"query": {"type": "string", "description": "A valid SQL SELECT statement."}}

SQL dialect: DuckDB. Notes:
- Aggregates available natively: COUNT, SUM, AVG, MIN, MAX, STDDEV, VARIANCE, MEDIAN, QUANTILE.
- Only SELECT statements are permitted.

To use a tool, respond with a JSON object (no prose, no markdown fences):
{"tool": "ds_2024_sales__run_query", "arguments": {"query": "SELECT ..."}}

To signal that you have enough information to answer, respond with:
FINAL ANSWER: <your complete answer here>
```

## Concurrency Model

- One query runs at a time per HTTP request, on its own daemon thread.
- Each run owns one event loop (`asyncio.run`) and one MCP pool keyed by `run_id`; the pool holds one `ClientSession` per attached source. Sessions are process-local and torn down at run end.
- No checkpointing in v0.1.
