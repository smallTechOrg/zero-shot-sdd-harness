# Architecture

## System Overview

The data analysis agent is a single-process FastAPI application. Users interact via a browser UI (Jinja2
templates). The primary entity is an **MCP server**, backed **1:1 by a dataset** addressed by a URI:
an internal dataset is a **directory of related Parquet files** (one uploaded CSV → one Parquet file →
one **table**); an external dataset is a **database** (PostgreSQL, BETA). A dataset is not a separate
entity — its URI, type, physical-table catalog, and a generated dataset JSONSchema all live in the MCP
server's **metadata**. An MCP server has three kinds of **children** (per the MCP 2025-06-18 spec):
**tools**, **resources**, and **prompts**, stored as typed rows and **LLM-generated by a sync pipeline**.

A **session** is the long-lived agent context over one or more MCP servers. The first time a session is
queried, the app builds a **per-session MCP pool** — one in-process FastMCP server per attached MCP-server
entity, exposing a **generic read-only-SQL tool** over that server's tables (Phase A). The pool is
**reused by every subsequent query**. Each query is one LangGraph ReAct run acting as an MCP **client**:
`plan_action → execute_action (call_tool) → finalize`, looping until the LLM signals a final answer. Tool
calls are **single-level** — `{"tool":"<server>","arguments":{"query":"SELECT …"}}`. The agent's memory
is **durable per session** via a LangGraph `SqliteSaver` checkpointer keyed by `thread_id = session_id`.

## Two MCP surfaces (keep separate)

There are **two distinct MCP surfaces**, intentionally not shared:

1. **The agent pool** (`tools/mcp/pool.py` + `tools/mcp/server.py`): an in-process FastMCP server per
   session per attached MCP-server entity, exposing a generic SQL tool. This is what the LangGraph agent
   calls. (Phase B also surfaces the generated GET-API tools here — hybrid.)
2. **The DB-backed JSON-RPC dispatcher** (`tools/mcp/dispatch.py`): answers `POST /mcpserver/{id}` from
   the stored tools/resources/prompts rows. This is the standards-compliant MCP surface for the UI and
   external MCP clients. It does **not** go through FastMCP — FastMCP cannot represent custom
   `inputSchema`/resources/prompts; the dispatcher reads the rows and executes canned SELECTs directly.

Key decisions: **(1)** a server's capabilities are **generated and stored as rows**, regenerated by the
sync pipeline — adding a dataset *type* means writing a new connector; **(2)** MCP is both the agent↔tool
transport (in-process) and the public protocol surface (`POST /mcpserver/{id}`); the LLM↔agent ReAct
protocol stays hand-rolled; **(3)** all MCP/tool code lives under `tools/`, connectors behind a
`DatasetConnector` seam in `tools/connectors/`, the sync pipeline in `tools/sync/`.

## Component Map

```
Browser (HTML form)
    ↓ POST /mcpserver/upload  (CSV → Parquet, returns data URI)
    ↓ POST /mcpserver         (create from URI+type → connection-check → run sync)
    ↓ POST /mcpserver/{id}/sync          (re-run 5-stage pipeline; versioned soft-delete apply)
    ↓ POST /mcpserver/{id}               (MCP JSON-RPC: tools/resources/prompts list/read/get/call)
    ↓ POST /sessions          |  POST /sessions/{id}/query
FastAPI (uvicorn, sync endpoints)
    │  upload : dataset_name + CSV → Parquet (FileIngester) → returns parquet:///{name}  (no entity)
    │  create : read dataset dir → physical_tables_json → connection_check() BEFORE commit
    │           → McpServer row → run_sync() generates tools/resources/prompts (1:1 URI enforced)
    │  sync   : 5-stage LLM pipeline → transactional apply (insert/update/SOFT-DELETE) → version++
    │  {id}   : JSON-RPC dispatch over stored capability rows (tools/mcp/dispatch.py)
    │  session: create SessionRow + links; best-effort warm the session pool
    │  query  : create QueryRecord + AgentRun, spawn a daemon thread
    │
    └─► Pipeline thread → SessionPoolManager.acquire(session_id)   (lazy build, reused)
                        → per-session lock → asyncio.run:
                            AsyncSqliteSaver(checkpoint_db) → build_graph().compile(checkpointer)
                            → ainvoke(input, thread_id=session_id)
            ├── plan_action    (reads servers/columns/memory; LLM picks a server + SQL, or FINAL ANSWER)
            ├── execute_action (MCP client call_tool(server) → DuckDB SELECT)
            └── finalize       (persist QueryRecord; write the full `conversation` list)
                ↓
        SQLite (metadata) + checkpoint SQLite (memory) + Parquet dirs / external DB (DuckDB)
```

## MCP Layer (agent pool)

```
        tools/mcp/pool.py — SessionPoolManager (the ONLY importer of mcp.shared.memory)
        ┌────────────────────────────────────────────────────────────────┐
 agent  │  per session_id (held, reused across queries; LRU/idle evicted):│
        │     1 FastMCP server PER MCP-SERVER ENTITY + DuckDB conn + lock  │
        │  per call (transient): ClientSession ─► FastMCP(server_i) ─► DuckDB
        │                          └─ query  (generic read-only SELECT over all the server's tables)
        └────────────────────────────────────────────────────────────────┘
   get_connector(server, tables).build_server() — tools/connectors/* + tools/mcp/server.py
```

- **Transport:** in-process / in-memory (`create_connected_server_and_client_session`). Sessions are
  **transient** (opened/closed within a single graph node); servers + DuckDB connections persist for the
  **session** and are reused by every query.
- **One FastMCP server per MCP-server entity**, exposing a **generic `query` tool** that runs read-only
  SELECT over all of that entity's tables (registered as DuckDB views in one connection → within-server
  JOINs work). Cross-server joins are not possible (separate servers) — the agent composes those across
  ReAct iterations.
- **Addressing:** single-level — `call_tool(session_id, server_name, arguments)` routes to that server's
  `query` tool.
- **Connector seam:** `tools/connectors/` provides `DatasetURI` (`uri.py`), the `DatasetConnector`
  Protocol + `DatasetConnectionError` + `get_connector(...)` factory (`base.py`), `ParquetConnector`
  (`parquet.py`), and `PostgresConnector` (`postgres.py`, BETA, flag-gated). Shared SQL helpers
  (`build_server`, `_run_select`, `_run_select_params`, `DEFAULT_MAX_ROWS`) stay in `tools/mcp/server.py`.
- **Lifecycle:** lazy build on first query; reused; **idle/LRU eviction**; closed on session delete +
  app shutdown; invalidated when a server changes (create/sync/delete). `close(session_id)` is
  **lock-safe** (acquires the per-session lock before closing) so a change never closes a DuckDB conn
  mid-query. Queries on one session are **serialized** by a per-session lock.
- **Isolation seam:** every MCP import lives in `tools/mcp/`. The `mcp` SDK is pinned `==1.28.0`.

## Layers

| Layer | Responsibility |
|-------|----------------|
| API (FastAPI) | HTTP routing; `mcpserver` (upload/create/sync/JSON-RPC dispatch/delete), `sessions`, `queries`; connection-check before commit; warms/closes session pools on server change |
| Ingestion | CSV/XLSX/JSON → Parquet; schema + row-count extraction (`tools/ingester.py`) |
| Connectors | `DatasetConnector` seam: `DatasetURI`, `get_connector` factory, `ParquetConnector`, `PostgresConnector` (BETA); `connection_check` + `discover_tables` + `build_server` (`tools/connectors/`) |
| Sync pipeline | 5-stage LLM generation of tools/resources/prompts + transactional versioned soft-delete apply (`tools/sync/`) |
| MCP server (agent) | Per-entity `FastMCP`, generic read-only `query` tool via DuckDB views; `build_server`/`_run_select` (`tools/mcp/server.py`) |
| MCP dispatch (public) | DB-backed JSON-RPC over stored rows: tools/resources/prompts list+read/get+call, cursor pagination (`tools/mcp/dispatch.py`) |
| Session pool manager | Build/cache/evict per-session pools; single-level routing, lock-safe close, per-session lock (`tools/mcp/pool.py`) |
| Graph (LangGraph) | Async ReAct pipeline: plan → execute → loop → finalize (no `load_data`) |
| Memory | Durable per-session checkpointer (`SqliteSaver`, `thread_id = session_id`) |
| LLM (OpenRouter) | Chat completions; falls back to stub when key not set |
| DB (SQLAlchemy + SQLite) | Persistence of MCP servers + capability rows, sessions, query history |
| Templates (Jinja2) | Server-rendered HTML |

## Data Flow

1. **Upload (auxiliary):** the form supplies a `dataset_name` + a CSV. The CSV → Parquet (`FileIngester`)
   at `{datasets_dir}/{slug(name)}/{table}.parquet`. Returns the data URI `parquet:///{name}`. No entity
   is created here. (UI: a *new* dataset → `POST /mcpserver`; an *existing* dataset → Phase B
   `resources/add`.)
2. **Create server (`POST /mcpserver`):** take a URI + type; reject a duplicate URI/name (1:1); read the
   dataset dir → build `physical_tables_json`; **`connection_check()` runs before commit** (a broken
   server is never persisted); insert the `McpServer` row; then **`run_sync()`** generates the
   tools/resources/prompts.
3. **Sync (`POST /mcpserver/{id}/sync`):** run the 5-stage pipeline; **transactionally** apply the diff
   (match→update, new→insert, missing→**soft-delete**), bump `version`, set title/description/schema/
   `last_synced_at`/`last_sync_status`; close affected session pools.
4. **MCP JSON-RPC (`POST /mcpserver/{id}`):** dispatch on `method` over the stored capability rows
   (`tools/mcp/dispatch.py`).
5. **New session:** create `Session` + `SessionMcpServer` links; **best-effort warm** the pool.
6. **Query:** create `QueryRecord` + `AgentRun`; spawn a daemon thread → `run_pipeline()`.
7. **Per-query run:** acquire the session pool (lazy-build if evicted) + the per-session lock; inside one
   `asyncio.run`, open the checkpointer and `ainvoke` with `thread_id = session_id`:
   - `plan_action`: read the servers snapshot (name, description, tables, columns) + the durable
     `conversation`; ask the LLM for a single-level `{"tool","arguments"}` call or `FINAL ANSWER:`.
   - `execute_action`: `manager.call_tool(session_id, tool, arguments)` → DuckDB `SELECT` (may JOIN the
     server's sibling tables); append result/error to the per-query `action_history`.
   - loop until `FINAL ANSWER:` or max iterations; `finalize` persists the record and writes the full
     `conversation` list (memory). **The pool is not closed here.**
8. **Result:** redirect to the session page (polls `/status`); the answer renders inline.

## External Dependencies

| Dependency | Purpose | Failure Mode |
|------------|---------|--------------|
| OpenRouter (Gemini 2.5 Flash) | NL reasoning / planning / 5-stage sync | Falls back to stub — "(stub mode)" |
| `mcp` SDK (in-process) | Agent↔tool protocol | Server build / session failure → fatal for that run |
| DuckDB | Read-only SQL over Parquet + external `ATTACH` | SQL errors recoverable; missing Parquet fatal |
| `langgraph-checkpoint-sqlite` | Durable per-session memory | Checkpoint DB unwritable → memory disabled / run error (degrade clearly) |
| SQLite | Metadata + checkpoint stores | App fails to start if unwritable |
| Local filesystem | Per-dataset Parquet directories (`{datasets_dir}/{slug(name)}/`) | Upload fails with a user-visible error |
| External PostgreSQL (BETA) | Live SQL datasets, gated by `DATAANALYSIS_ENABLE_EXTERNAL_DATASETS` (default off → create returns 501) | connection-check failure → sanitized `connection_error`; never persisted broken |

## Concurrency, Async & Memory Model

- FastAPI endpoints stay **sync**; a query runs on a **daemon thread**; `run_pipeline()` owns one
  `asyncio.run` per query.
- LangGraph runs **each node in its own task** → MCP `ClientSession`s are **transient per node**; the
  manager holds only plain objects (servers + DuckDB conns) across nodes and queries.
- **Per-session serialization:** a per-session `threading.Lock` wraps each query (shared DuckDB conn).
  Eviction skips locked (in-use) sessions.
- **Memory:** `AsyncSqliteSaver` (file-backed) keyed by `thread_id = session_id`; a fresh saver is opened
  inside each query's loop. **`conversation` is a plain last-value channel** — `finalize` writes the
  **full** list each time (idempotent under checkpoint replay). It is **not** a reducer: an `operator.add`
  reducer double-appends when the checkpoint is restored on the next query.
- **Constraints (non-negotiable):** no LangGraph parallel fan-out; never span an MCP `ClientSession`
  across nodes; never wrap MCP calls in `anyio.to_thread`; never make `conversation` a reducer channel.

## Deployment Model

Local single-user service: `uv run python -m data_analysis_agent` on port 8001. Single process — MCP
servers are in-memory; the metadata DB, the checkpoint DB, and Parquet files are the only on-disk state.
