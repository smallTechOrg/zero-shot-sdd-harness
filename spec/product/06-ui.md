# UI

## UI Type

Web browser — server-rendered HTML via Jinja2 templates. Minimal vanilla CSS. No JavaScript framework in
v0.1 (a little inline JS for the upload→create two-step and inline confirms).

## Views / Screens

### Screen: Home

**Route:** `GET /`

**Purpose:** Primary entry point showing both first-level entities — **MCP servers** and **Sessions** —
side by side. Manage servers and create/open/delete sessions from one screen.

**Layout:** Two columns (stack on mobile).

**Left column — MCP servers:**
- Section title: "MCP Servers"
- "Add a dataset" inline form:
  - **Dataset name** field (required)
  - Type chooser: "Upload CSV" (default) or "Connect a database"
    - "Upload CSV" → file input. On submit the page first calls `POST /mcpserver/upload` (CSV → Parquet,
      returns the data URI), then `POST /mcpserver` with that URI to create the server (which runs sync).
    - "Connect a database" → `dataset_uri` text field; shown **only when external datasets are enabled**
      (`enable_external_datasets`), otherwise hidden. Submits straight to `POST /mcpserver`.
  - Submit button
- List of MCP-server cards, each showing:
  - Name + LLM **title**, type badge (`parquet` / `postgresql`), **version**
  - "{n} table(s)" and counts of active **tools / resources / prompts**
  - Credential-free URI (`DatasetURI.display()` — never user/password)
  - `connection_error` badge (only on error) and `last_sync_status`
  - `last_synced_at` timestamp
  - **Sync** button → `POST /mcpserver/{id}/sync`
  - **Open** (detail) → `GET /mcpserver/{id}`
  - **Add CSV** (parquet only) → uploads a CSV then re-syncs the server (Phase B: `resources/add`)
  - Delete button with inline confirmation
- Empty state when there are no servers

**Right column — Sessions:**
- Section title: "Sessions"
- "New Session" button → inline form: optional name + a checkbox list of MCP servers (pick ≥1; field name
  `mcp_server_ids`) + "Create Session" + "Cancel"
- Session cards: name + date, server chips (each shows the server name + table count), query count,
  "Open" → `GET /sessions/{id}`, Delete with inline confirm
- Empty state when there are no sessions

**Actions:** create server (upload CSV → create, or connect a DB) · sync server · open server detail ·
delete server · create/open/delete session.

### Screen: MCP Server detail

**Route:** `GET /mcpserver/{id}`

**Purpose:** Inspect a generated MCP server.

**Key elements:**
- Header: name + title, type badge, credential-free URI, version, `last_sync_status`, `last_synced_at`
- **Dataset** section: the physical tables (name, row count, columns) from `physical_tables_json`
- **Tools** list: name, title, description, parameters (from `inputSchema`)
- **Resources** list: name/uri, kind (schema / primary_entity / secondary_entity), description
- **Prompts** list: name, title, description, arguments
- **Sync** button → `POST /mcpserver/{id}/sync` (regenerates; version bumps; soft-deletes only)
- Stub-mode banner

---

### Screen: Session (Chat)

**Route:** `GET /sessions/{session_id}`

**Purpose:** The main interaction screen.

**Key elements:**
- Breadcrumb: Home › Session name
- Header: session name + a chip per attached MCP server (name + table count)
- "Ask a question" card (top): text input, submit, loading overlay
- Past Q&A cards inline (newest first): question, answer, SQL iterations, token usage, cost; collapsible
  agent-reasoning trace showing each step's `tool` (server) and SQL; `?new=` card highlighted
- Stub-mode banner

**Actions:** submit question → `POST /sessions/{session_id}/query`

---

### Screen: Error

**Route:** rendered inline on 4xx/5xx — error message + "Back" link.

## Error States

- Pipeline errors: render `error.html` — never a bare 500
- Upload/create errors: flash message on home (credential-free)
- 404: render `error.html` with a "not found" message

## Tech Stack

Jinja2 + FastAPI. Minimal inline CSS, minimal inline JS (upload→create step, inline confirms). Stub-mode
banner via the `llm_provider` context variable.
