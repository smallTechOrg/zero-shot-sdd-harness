# UI

> **Spec status:** Filled in for the **Senior Data Analyst Agent** (`data-analyst`), v0.1. Last updated 2026-06-22.

---

## UI Type

**Local web app** — FastAPI serving server-rendered HTML pages. Single user. v0.1 renders **text and tables only**; charts/visualizations are deferred (see `01-vision.md` → Future Phases).

## Views / Screens

### Screen: Home / Sessions

**Purpose:** Create a new session or reopen an existing one (sessions persist across restarts).

**Key elements:**
- List of existing sessions (name + last-updated).
- "New session" form (name field).

**Actions available:**
- Create a session → navigates to its session view.
- Open an existing session.

---

### Screen: Session View

The single working screen. Three regions.

**Purpose:** Upload datasets, ask natural-language questions, read SQL-backed answers, and inspect the audit log — all for one session.

**Key elements:**
- **Left panel — Datasets:**
  - Upload control (CSV/Parquet), with optional display-name field.
  - List of registered datasets: name, format, row count. (Optionally expand to show cached schema columns.)
- **Center panel — Chat:**
  - Scrollable conversation history (persisted messages).
  - For each assistant turn: the NL **answer text**, a rendered **result table** (columns + rows from DuckDB), and a way to reveal the **generated SQL** (collapsible).
  - Input box to type a natural-language question + send.
- **Right/secondary — Audit panel (tab or side panel):**
  - Reverse-chronological audit entries for this session: timestamp, NL prompt, generated SQL, row count, duration, status (success/error).

**Actions available:**
- Upload a dataset (capability 01).
- Ask a question (capability 02) → answer + table appended to chat.
- Toggle/reveal generated SQL on an answer.
- Open the audit panel (capability 03).

## Error States

- **Stub LLM banner (mandatory):** when no API key is configured and the provider is the stub, a visible banner appears on **every page** stating the agent is running in offline/stub mode (per the boilerplate's non-negotiable rule on signalling stubs).
- **Upload errors:** unsupported/corrupt file → inline error near the upload control; no dataset added.
- **Empty-state guards:** asking with no datasets in the session → inline prompt to upload first (matches API `400`).
- **Query failure:** if the agent fails (bad SQL after retry / LLM error), the chat shows a friendly error message (not a stack trace) and the failure appears as an `error` entry in the audit panel.
- **Loading state:** while a question is processing, the send control shows a pending/disabled state.
