# Capability 4: Generate & Serve MCP Capabilities (Sync Pipeline + JSON-RPC Surface)

## Overview

Every MCP server's **tools, resources, and prompts** are LLM-generated by a **5-stage sync pipeline** and
served over a standards-compliant **JSON-RPC endpoint** (`POST /mcpserver/{id}`). Sync is **incremental**
(the LLM is shown the existing capabilities), **versioned** (each apply bumps `McpServer.version`), and
**soft-delete-only** (a suggestion can never irreversibly delete a capability).

## The Sync Pipeline (`tools/sync/`)

`run_sync(server_id) -> SyncResult`. Each stage builds a prompt carrying (a) the **first 5 rows** of each
physical table and (b) the server's **existing** capabilities (so the LLM edits incrementally), calls the
LLM, parses JSON, and **falls back to a deterministic template on any failure** (sync never hard-fails).
Each stage injects a distinct node tag so the stub branches deterministically (see `ai-agents.md` §8).

| Stage | Tag | Produces | Persists to |
|------|-----|----------|-------------|
| a. title + description | `<node:sync_title>` | `{title, description}` | `mcp_servers.title`, `.description` |
| b. dataset JSONSchema | `<node:sync_schema>` | `{tables:{…}, relationships:[{from,to,on}]}` | `mcp_servers.dataset_schema_json` + a `kind='schema'` resource |
| c. entities → resources | `<node:sync_entities>` | `{entities:[{name,title,description,kind,uri,mime_type}]}` | `mcp_resources` (primary/secondary) |
| d. GET-APIs → tools | `<node:sync_tools>` | `{tools:[{name,title,description,input_schema,sql_template,output_schema?}]}` | `mcp_tools` |
| e. prompts | `<node:sync_prompts>` | `{prompts:[{name,title,description,arguments,template}]}` | `mcp_prompts` |

The cascade threads b → c → d → e: the schema feeds the entities; entities feed the tools; tools feed the
prompts. (This is the dependency chain that the Phase-B partial syncs follow in reverse.)

### Apply (transactional, versioned, soft-delete-only)

In **one transaction**:
1. Load the server's **active** capabilities (per type).
2. `new_version = server.version + 1`.
3. For each proposed item, match an existing active row by stable key (`name` for tools/prompts;
   `uri` or `kind`+name for resources): match → **update in place**; new → **insert** with
   `created_version = new_version`.
4. Any active row **absent** from the proposal → set `deleted_at` + `deleted_version = new_version`
   (**never** hard-delete).
5. **Validate** each generated `sql_template` compiles (run it with `LIMIT 0`); skip/flag a tool whose
   SQL doesn't compile rather than persisting a broken tool.
6. Set `server.version = new_version`, title/description/`dataset_schema_json`, `last_synced_at`,
   `last_sync_status` (`ok`/`partial`/`error`). Commit (rollback on any error → no half-applied sync).
7. Close the pools of sessions attached to this server (lock-safe).

### Stub determinism (offline)

The stub provider has one branch per stage tag. Each echoes identifiers parsed from its prompt (table
names → entities → tool names → prompt names) so the cascade is self-consistent, and emits a valid
**zero-parameter** SELECT (e.g. `SELECT COUNT(*) AS n FROM {table}`) for tools so `tools/call` and the
agent both work offline. A test asserts every stage tag is handled (a missing branch yields a garbage
sync).

## The JSON-RPC Surface (`POST /mcpserver/{id}`)

JSON-RPC 2.0 over the server's **active** capability rows (`tools/mcp/dispatch.py` — a DB-backed
dispatcher, **not** FastMCP). Read methods (Phase A):

- `tools/list {cursor?}` → MCP Tool objects + `nextCursor?`
- `tools/call {name, arguments}` → bind `arguments` into `sql_template` (DuckDB **parameter binding**),
  run via the read-only guard → `CallToolResult {content, structuredContent?, isError}`. SQL failure →
  `isError:true`; unknown tool → JSON-RPC `-32602`.
- `resources/list {cursor?}` / `resources/read {uri}` → `{contents:[{uri,mimeType,text}]}` (`kind='schema'`
  returns `dataset_schema_json`).
- `prompts/list {cursor?}` / `prompts/get {name, arguments?}` → `{description, messages}`.
- Unknown method → `-32601`; bad params/cursor → `-32602`.

**Pagination:** opaque keyset cursor (base64 of the last `(created_at, id)`); `nextCursor` omitted on the
last page; page size `mcp_list_page_size` (default 50).

## Phase B — Granular mutation methods (specced; deferred impl)

Additional JSON-RPC methods on `POST /mcpserver/{id}`, each followed by a **partial sync** that re-asks the
LLM only for the downstream capabilities (soft-delete-only, version-bumping, transactional — same apply
rules):

| Method | Effect | Partial-sync cascade |
|--------|--------|----------------------|
| `tools/add` | Insert a tool | → re-generate **prompts** |
| `tools/update` | Update a tool | → re-generate **prompts** |
| `prompts/add` | Insert a prompt | (no cascade) |
| `prompts/update` | Update a prompt | (no cascade) |
| `resources/add` | Insert a resource (new entity) | → re-generate **tools**, then **prompts** |
| `resources/update` | Update a resource | → re-generate **tools**, then **prompts** |

Cascade chain: **resources (entities) → tools (GET-APIs over entities) → prompts (over tools)**. Adding a
CSV to an existing dataset = a new physical table = `resources/add` (a new entity), which cascades a
partial sync of tools + prompts. Until implemented these methods return `-32601`.

## Phase B — Hybrid agent consumption (specced; deferred impl)

The session pool will also surface a server's generated GET-API tools (alongside the generic `query`
tool). The agent prefers a matching generated tool (calling it by name with the declared parameters) and
falls back to the generic SQL tool for anything not covered. This requires the pool/planning/execution to
present both surfaces; the generic-SQL path (Phase A) remains the fallback.

## Success Criteria

- Creating a server runs sync and yields ≥1 tool, ≥1 resource (the schema), ≥1 prompt — offline (stub).
- Re-syncing bumps the version, updates matched capabilities, and **soft-deletes** (never removes) dropped
  ones; a re-added name inserts a fresh active row without colliding with the tombstone.
- `POST /mcpserver/{id}` answers all six read methods with correct shapes, cursor pagination, and the
  `-32601`/`-32602`/`isError` error mapping.
- A generated tool's SQL is validated at apply; `tools/call` binds parameters safely (no injection).
