# FR-001 — Data Analyst Agent

**Date:** 2026-06-22
**Author:** researcher
**Status:** approved

---

## Problem Statement

Technical and semi-technical analysts spend significant time writing SQL manually or
switching between tools to explore structured datasets. They lack a conversational interface
that behaves like a senior data analyst: clarifying ambiguous questions, generating and
executing SQL, explaining results, and proactively suggesting follow-up angles. This FR
describes a local-first web application that closes that gap — a natural-language data
analyst agent backed by DuckDB and Google Gemini 2.5 Flash.

## Target Users

| User | Context | Primary need |
|------|---------|--------------|
| Data analyst | Works with CSV/JSON exports from internal tools; wants fast answers without writing SQL | Ask questions in plain English and get tables, charts, and explanations |
| Developer / ML engineer | Prototyping data pipelines; needs rapid data exploration without standing up a full analytics stack | Upload a file, query it immediately, iterate quickly |

## Success Criteria (EARS)

### Dataset management

- [ ] WHEN a user uploads a CSV or JSON file and provides a dataset name, the system SHALL
  register the dataset in DuckDB and confirm the name, row count, and column list within the
  same HTTP response.

- [ ] WHEN a user lists their datasets, the system SHALL return the name, file format, row
  count, and upload timestamp for every registered dataset in the current session.

- [ ] WHILE a session is active, the system SHALL make all previously uploaded datasets
  available for cross-dataset queries without re-uploading.

### Natural-language querying

- [ ] WHEN a user sends a natural-language question referencing one or more registered
  datasets, the system SHALL translate the question to SQL, execute it against DuckDB, and
  return the result set within 30 seconds.

- [ ] WHEN the generated SQL would scan a file larger than 100 MB, the system SHALL warn the
  user before executing and proceed only on confirmation.

- [ ] IF the user does not confirm a large-file execution, THEN the system SHALL abort the
  query and return a message indicating the query was cancelled without executing SQL.

- [ ] IF the user's question is ambiguous (e.g. the column referenced is not uniquely
  identifiable), THEN the analyst agent SHALL ask one clarifying question before generating
  SQL rather than guessing.

### Rich response output

- [ ] WHEN a query returns tabular data, the system SHALL include a Markdown table in the API
  response and the web UI SHALL display the result in a tabular format.

- [ ] WHEN the query result set contains at least one numeric column and at least one
  categorical or date column, the system SHALL return a valid Chart.js configuration object
  alongside the table and the web UI SHALL render it as an interactive chart.

- [ ] WHEN the analyst agent returns a dashboard response (multiple charts), the web UI
  SHALL render each chart in a grid layout without page reload.

### Analyst workflow simulation

- [ ] WHEN the analyst agent produces a result, it SHALL append at least one non-empty
  follow-up question string that references at least one column name from the result set.

- [ ] WHEN the user asks the analyst agent to explain a result, the system SHALL respond in
  plain English describing what the data shows and any notable patterns, without exposing
  raw SQL in the explanation unless the user explicitly requests it.

### Token economy

- [ ] The system SHALL send only the dataset schema (column names and types) — not raw data
  rows — to the LLM when generating SQL, keeping prompt token count proportional to schema
  size rather than data volume.

- [ ] The system SHALL log prompt token count and completion token count for every LLM call
  to the audit log; the web UI SHALL display a running session total.

### Session persistence

- [ ] WHILE the server is running, all session state (uploaded datasets, conversation
  history, audit log) SHALL persist across HTTP requests using the session ID provided by
  the client.

- [ ] WHEN the server restarts and the client reconnects with a valid session ID, the system
  SHALL restore all datasets registered in that session and return the full ordered
  conversation history as a JSON array in the session-detail endpoint so the user can
  continue without re-uploading files.

### Audit logging

- [ ] The system SHALL write one audit log entry per SQL execution containing: timestamp,
  session_id, natural_language_query, generated_sql, rows_returned, duration_ms,
  prompt_tokens, completion_tokens.

- [ ] WHEN a user requests the audit log for their session, the system SHALL return all
  entries as JSON ordered by timestamp ascending.

## Non-Goals

- Authentication or multi-user access control — not required for this milestone.
- Cloud or production deployment — local demo only; Render deployment is a separate request.
- Real-time or streaming data sources — batch file upload only.
- Parquet or Avro file support — deferred; CSV and JSON are the only formats at launch.
- Complex dashboard builder with drag-and-drop UI — chart layout is fixed grid only.
- Multi-tenant data isolation — all data is session-scoped, single user.

## Key Constraints

| Constraint | Value | Reason |
|-----------|-------|--------|
| Language | Python 3.12+ with `uv` | Project default; fast onboarding |
| Backend framework | FastAPI | Async, typed, matches recipe scaffold |
| Agent framework | LangGraph | Reasoning loop with clarify / query / explain steps |
| LLM | Google Gemini 2.5 Flash (`ANALYST_LLM_MODEL=gemini-2.5-flash`) | Token-efficient; user's approved provider |
| Analytics database | DuckDB (in-process, file-backed) | Columnar; native CSV/JSON ingestion; no server to run |
| Metadata / audit store | SQLite (spine alongside DuckDB) | Lightweight relational store for session metadata and audit log |
| Frontend | Next.js 15 | Required for Chart.js integration and dashboard rendering |
| Input file formats | CSV, JSON only | Parquet deferred |
| File size ceiling | 100 MB per file | Analyst-scale; larger files require explicit warning |
| Token economy | Schema-only prompts; token counts logged | Primary constraint; must not send raw rows to LLM |
| Session persistence | Sessions survive server restart | Stored on disk (DuckDB file + SQLite) keyed by session ID |
| Auth | None | Single-user local demo |
| API key | `GEMINI_API_KEY` — present in environment | Required from Phase 3 onward; Phase 2 gate uses stub mode |
| Deploy target | Local only (`localhost:8001`) | Render on explicit future request |
| Port | 8001 | Permanent harness rule |
