# Data Model

## Storage Technology

SQLite, file at `data/agent.db`. Chosen because: (1) the spec requires local-only storage; (2) no concurrency scaling requirements; (3) no cloud DB dependency. Two categories of tables exist in the same file:

1. **Metadata tables** — `upload_sessions`, `query_runs` — created by Alembic migrations, always present.
2. **Dynamic data tables** — one per CSV upload, created at upload time via raw DDL (`CREATE TABLE IF NOT EXISTS ...`), never altered, never dropped by the application (manual cleanup only).

---

## Entities

### Entity: `UploadSession`

Represents a single CSV file upload. Created when the user uploads a CSV. Each row tracks the mapping from `session_id` to the dynamic SQLite table that holds the uploaded data.

SQLAlchemy model: `src/db/models.py` → `class UploadSession`
Table name: `upload_sessions`

| Field | SQLAlchemy Type | Nullable | Description |
|-------|-----------------|----------|-------------|
| `id` | `Text` PK | No | UUID (string). Generated by `uuid4()` at insert time. Also used as the `session_id` returned to the client. |
| `table_name` | `Text` unique | No | Derived dynamic table name, e.g. `sales_data_a3f7b2c1`. Used by the graph pipeline to locate the data. |
| `original_filename` | `Text` | No | The filename as uploaded by the user, stored for display purposes only. |
| `row_count` | `Integer` | No | Number of data rows (excluding header) successfully inserted. |
| `col_count` | `Integer` | No | Number of columns in the CSV. |
| `schema_json` | `Text` | No | JSON string: `[{"column": "name", "type": "TEXT"}, ...]`. Stored to avoid re-running `PRAGMA table_info` on every query. |
| `created_at` | `TIMESTAMP(timezone=True)` | No | UTC timestamp, set at insert, never updated. |

---

### Entity: `QueryRun`

Represents one natural-language question and its full answer (SQL, chart spec, insight). Created by `POST /query` after the pipeline completes (success or failure).

SQLAlchemy model: `src/db/models.py` → `class QueryRun`
Table name: `query_runs`

| Field | SQLAlchemy Type | Nullable | Description |
|-------|-----------------|----------|-------------|
| `id` | `Text` PK | No | UUID (string). Returned to the client as `query_run_id`. |
| `session_id` | `Text` FK → `upload_sessions.id` | No | The session this query belongs to. |
| `question` | `Text` | No | The natural-language question as submitted. |
| `sql` | `Text` | Yes | The generated SELECT query. NULL if `sql_generation` failed. |
| `chart_spec` | `Text` | Yes | JSON string of the Recharts chart spec. NULL if `chart_selection` was not reached. |
| `insight` | `Text` | Yes | Plain-English insight paragraph. NULL if `insight_generation` was not reached. |
| `status` | `Text` | No | `"pending"` → `"completed"` or `"failed"`. |
| `error` | `Text` | Yes | Human-readable error message if `status = "failed"`. NULL otherwise. |
| `created_at` | `TIMESTAMP(timezone=True)` | No | UTC timestamp, set at insert. |
| `updated_at` | `TIMESTAMP(timezone=True)` | No | UTC timestamp, set at insert, updated on status change. |

---

### Dynamic Data Tables (not ORM-managed)

Created by `POST /upload` via raw DDL. Not tracked by SQLAlchemy ORM or Alembic. Naming: `{slug}_{session_id[:8]}`.

Schema is inferred from the CSV: columns are `TEXT`, `INTEGER`, or `REAL` based on the type inference rules in `spec/capabilities/csv-upload.md`. All values are inserted as the inferred native type; empty cells become `NULL`.

Example:
```sql
CREATE TABLE IF NOT EXISTS sales_data_a3f7b2c1 (
    product_name TEXT,
    quantity     INTEGER,
    revenue      REAL,
    sale_date    TEXT
);
```

These tables are never mutated after the initial `INSERT` and are never explicitly dropped by the application.

---

## Relationships

```
UploadSession (1) ──── (many) QueryRun
   via QueryRun.session_id → UploadSession.id
```

Each `UploadSession` has zero or more `QueryRun` records. Each `QueryRun` belongs to exactly one `UploadSession`.

The dynamic data table referenced by `UploadSession.table_name` is implicitly related to the session but is not an ORM entity.

---

## Data Lifecycle

| Event | Action |
|-------|--------|
| `POST /upload` succeeds | `UploadSession` row inserted; dynamic table created and populated. |
| `POST /query` called | `QueryRun` row inserted with `status="pending"`. |
| Pipeline succeeds | `QueryRun` updated: `status="completed"`, `sql`, `chart_spec`, `insight` populated. |
| Pipeline fails | `QueryRun` updated: `status="failed"`, `error` populated. |
| Phase 2: pin added | `QueryRun.pinned` set to `true` (Alembic migration adds column). |
| Application restart | No automatic cleanup. Data persists across restarts. |
| Manual cleanup | Dynamic tables and their `UploadSession` records must be deleted manually (no TTL in scope). |

---

## Sensitive Data

- `QueryRun.question` may contain business-sensitive data from the user's domain. It is stored unencrypted in SQLite.
- Dynamic table contents are the CSV data itself, stored unencrypted. Users are responsible for not uploading PII-containing CSVs in Phase 1 (no data-at-rest encryption in scope).
- No API keys or credentials are stored in the database. They live exclusively in `.env`.
- SQL generated by the LLM is logged in truncated form (200 chars max) but never logged in full to avoid PII leakage from column values in WHERE clauses.
