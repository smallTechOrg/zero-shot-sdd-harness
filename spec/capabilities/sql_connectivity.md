# Capability: SQL Connectivity

> **Phase 2: Not yet implemented.** This capability is a design stub. No code exists for it yet. It will be built in Phase 2 after Phase 1 (CSV Analysis) is complete and human-approved. Do not reference any of the interfaces below as existing.

---

## What It Does

Connects to a local SQLite, PostgreSQL, or MySQL database by connection URL, lists its tables, and answers natural-language questions by generating and executing SQL queries against the live database.

## Inputs

| Input | Type | Source | Required |
|-------|------|--------|----------|
| db_url | string (connection URL) | User input via `POST /connections` | Yes |
| db_type | enum: `sqlite` \| `postgres` \| `mysql` | Inferred from URL scheme | Yes |
| question | string | User text input via `POST /analyses` | Yes |
| connection_id | string (UUID) | Returned by `POST /connections` | Yes |

## Outputs

| Output | Type | Destination |
|--------|------|-------------|
| schema_summary | string | `POST /connections` response body — table names + column names + types |
| answer_text | string (plain English) | `POST /analyses` response body |
| sql_query | string | `POST /analyses` response body (shown to user for transparency) |
| chart_json | string (Plotly JSON) or null | `POST /analyses` response body |

## External Calls

| System | Operation | On Failure |
|--------|-----------|------------|
| Local database | Introspect schema (tables, columns, dtypes) | Return HTTP 400 with connection error detail |
| Local database | Execute generated SQL query | Return HTTP 400 with SQL error detail; never retry destructive statements |
| Gemini API | Generate SQL query from natural language + schema | Set error; route to handle_error |
| Gemini API | Write plain-English answer from query results | Set error; route to handle_error |

## Business Rules

- Only `SELECT` statements are allowed; any generated SQL that is not a `SELECT` is rejected before execution.
- Connection URLs are stored in the `connections` table; the password component (if any) is masked in logs and API responses.
- The agent introspects the schema at connection time and caches it; the user can refresh the schema via `POST /connections/<id>/refresh`.
- Query results are limited to 1,000 rows before being passed to the answer node, to bound LLM prompt size.
- All database access is read-only; the connection must use a read-only user or the agent must enforce `SELECT`-only at the SQL parsing level.
- The database never leaves the local machine; only the schema summary and query results (truncated to 1,000 rows) are sent to Gemini.

## Success Criteria

- [ ] `POST /connections` with a valid SQLite URL returns HTTP 200 with a `connection_id` and a `schema_summary` listing all tables and their columns.
- [ ] `POST /analyses` with a `connection_id` and a question about a known table returns HTTP 200 with a correct plain-English answer and valid SQL in `sql_query`.
- [ ] A non-SELECT SQL statement generated by the LLM is rejected before execution and the response contains an error explaining why.
- [ ] A PostgreSQL connection URL with a wrong password returns HTTP 400 with a readable error message.
- [ ] Query results > 1,000 rows are truncated; the `answer_text` notes the truncation.
