## Planner notes

**Recipe selection:** copy `harness/recipes/python-fastapi-duckdb` and `harness/recipes/frontend-nextjs` to project root. The executor replaces every `APP_` / `app_` occurrence with `ANALYST_` / `analyst_` throughout the copied scaffold (env vars, config keys, comments) — this is the only renaming step required.

**Dependency edges:** Iter 1 (DuckDB registration + SQLite metadata) must land before Iter 2 (query loop) because the stub agent reads the registered schema to build its canned SQL. Iter 4 (persistence) depends on Iter 1's SQLite session model and Iter 2's conversation history schema. Frontend Iter 5 depends on the session ID model introduced in Iter 1; Iter 6 depends on both Iter 2 (query response shape including Markdown + Chart.js config) and Iter 5 (frontend scaffold) — the renderer and its data ship in the same iteration (Iter 6) to avoid the raw-`<pre>` carry-forward trap. Token fields in Iter 8 extend the audit log row from Iter 3 and the session-detail response from Iter 4.

**Stack-conditional traps to flag for executor:**
- Add `python-multipart` to dependencies (C-MULTIPART) — file upload will 500 without it.
- Register DuckDB tables with `CREATE OR REPLACE TABLE`, not views; re-register at lifespan startup (C-DUCKDB-VIEW) — views are connection-scoped and disappear on restart.
- Default `ANALYST_LLM_PROVIDER=stub`; add `ALLOW_MODEL_REQUESTS=False` in `tests/conftest.py` (C-STUB) — the offline gate must never silently hit the network.
- Use `react-markdown` + `remark-gfm` with `table`/`th`/`td` component overrides for Markdown table rendering, not `<pre>` (C-MD-RENDER).
- Chart.js via `react-chartjs-2`; use `dynamic(() => import(...), { ssr: false })` for the chart component to avoid SSR breakage (same principle as C-PLOTLY-SSR).
- Frontend reads `NEXT_PUBLIC_API_URL` with a fallback to `http://localhost:8001`; never hardcode (C-API-URL).
- Per-tab session ID: `crypto.randomUUID()` in a `useRef` initialiser with a `Math.random` fallback (C-SESSION-SCOPE).
- LLM calls use LangChain `init_chat_model` via `langchain-google-genai`; do **not** import `google-generativeai` (C-LLM-SDK). Strip markdown fences from Gemini JSON responses before `json.loads` (C-LLM-FENCE). Keep model name in `ANALYST_LLM_MODEL` env var, never hardcoded (C-LLM-MODEL).
- Guard `os.makedirs` on DuckDB file path — `os.path.dirname` of a bare filename returns `""` and crashes (C-DB-DIRNAME).
